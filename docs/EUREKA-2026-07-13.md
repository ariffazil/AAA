## 0. The Breakthrough

> **The agent is not the model. The agent is the closed constitutional loop that converts human intent into bounded, verifiable and metabolised consequence.**

> **Nothing is allowed to mean two things: identity is not authority, speech is not authentication, memory is not truth, execution is not success, telemetry is not proof, and intelligence is not sovereignty.**

## 0a. Two Canonical Corrections

### Correction 1 — Truth vs Authority

> **F13 is the final source of authority, not the source of absolute factual truth.**

Arif can decide whether the system may act, what values govern it, and where the final veto lies. But neither human sovereignty nor machine consensus can make a false geological measurement, broken hash chain, or failed deployment true.

```
Truth             ← reality + evidence
Meaning/Authority ← sovereign human
```

### Correction 2 — Entropy

> **The architecture does not mathematically guarantee ΔS < 0 across every execution.**

It creates the conditions for *controlled* entropy reduction. Individual actions may increase local entropy: failures occur, new information creates uncertainty, deployments produce state change. The architecture ensures that entropy is **measured, bounded, processed and converted into clearer structure**.

The accurate invariant:

```
ΔSᵤₙₐccₒᵤₙₜₑd ≤ 0
```

The system may encounter disorder, but it must not leave disorder unclassified, unauthorised, unobserved or unreceipted.

### 1.2 Governance Plane — Permission and Constraint

| Property | Value |

| **Invariant** | This plane decides what may proceed. It does **not** perform arbitrary production work. The governing sequence prevents one category from impersonating another. |

| Property | Value |
|----------|-------|

## 4. The 11-Step Chain

The permanent engineering principle:

> Classify first. Authorise second. Act third. Verify fourth. Remember fifth. Seal last. Cool always returns through governance.

```
1.  Understand the request
2.  Resolve the actor
3.  Classify the proposed action
4.  Calculate required authority
5.  Verify available authority
6.  Check evidence and consequences
7.  Issue a narrow capability
8.  Execute the exact action
9.  Verify the actual result
10. Update memory
11. Write the correct receipt
12. Cool the session and learn (through governance)
```

**Anti-pattern (before EUREKA):**
```
call tool → encounter gate → patch gate → encounter another gate → bypass gate
```

**Correct flow (after EUREKA):**
Each gate receives the facts it needs before enforcement begins.
