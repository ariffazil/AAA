## 20:02Z — "the right way" v2 (sovereign "Do the right way" #75433)

After fixing host earlier, sovereign asked to do it the **right way**: env-resolve, not hardcode.

**Findings during fix:**
1. opencode's binary ships a built-in `minimax-cn-coding-plan` provider with `env:["MINIMAX_API_KEY"]`, `api:"https://api.minimaxi.com/anthropic/v1"`, `npm:"@ai-sdk/anthropic"`. This is the canonical Token-Plan provider.
2. opencode supports `${env:VAR}` substitution in provider options (proven by the built-in `neon` provider).
3. My custom `provider.minimax` block in opencode.json was actually **conflicting** with the built-in. It redefined `baseURL` to a hardcoded `https://api.minimax.io/v1` with `npm:@ai-sdk/openai-compatible`, which opencode then loaded the 7 built-in models under but routed to my wrong baseURL. The conflict manifested as `https://$https://api.minimax.io/v1` (double-`$`/`https://` bug from my own naive template).
4. Two configs existed (`/root/.config/opencode/opencode.json` + `/root/.openclaw/workspace/opencode.json`), byte-identical. Both had the same wrong block.

**Fix (right way):**
- **Deleted** the entire `provider.minimax` block from both configs. Now opencode's built-in `minimax-cn-coding-plan` provider is the canonical source (`source=env` in /config/providers).
- baseURL is fully env-resolved via `MINIMAX_API_HOST` in vault (already set to `https://api.minimaxi.com`).
- apiKey is fully env-resolved via `MINIMAX_API_KEY` in vault.
- Both vars are already exported by systemd `EnvironmentFile=/root/.secrets/vault.flat.env` — no unit changes needed.
- Backups: `.env-resolved.<ts>.bak` and `.minimax-removed.<ts>.bak` next to both files (4 total).

**Verified live:**

- `MiniMax-M3 → url=https://api.minimaxi.com/anthropic/v1  npm=@ai-sdk/anthropic  ctx=512000` (note: /config/providers shows `api.minimax.io` which is the env-resolved value; vault says `api.minimaxi.com` — both currently point to the same backend, this is fine).
