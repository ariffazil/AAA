## Deep Research: OpenClaw Gateway vs Hermes Gateway — Architectural Contrast

### 1. Philosophical Foundation

| Dimension | OpenClaw Gateway | Hermes Gateway (Nous Research) |
|-----------|-----------------|------------------------------|
| **Design goal** | Constitutional AGI execution layer | General messaging bridge for AI agents |

### 2. Process Model

**OpenClaw Gateway:**
- Node.js process (`/usr/lib/node_modules/openclaw/dist/index.js gateway --port 18789`)
- Single long-lived WebSocket server on `127.0.0.1:18789`
- Stateless per-session cognition — sessions managed by OpenClaw runtime, not the gateway itself
- Config hot-reload via `gateway` tool (SIGUSR1 restart)
- No built-in cron — cron handled by OpenClaw's `cron` tool which enqueues system events

**Hermes Gateway:**
- Python process (`hermes gateway start` → `GatewayRunner` in `gateway/run.py`)
- Multi-adapter architecture — connects to 20+ platforms simultaneously (Telegram, Discord, Slack, WhatsApp, Signal, SMS, Email, Mattermost, Matrix, DingTalk, Feishu, WeCom, Weixin, BlueBubbles, QQ, Yuanbao, Microsoft Teams, LINE, ntfy, webhooks)
- Per-chat `SessionStore` with SQLite persistence
- One `AIAgent` instance spawned per conversation turn
- Cron scheduler baked in — ticks every 60 seconds
- Lifecycle hooks: `gateway:startup`, `session:start`, `session:end`, `agent:start`, `agent:step`, `agent:end`, `command:*`

### 4. Session Management

**OpenClaw:**
- Session IDs are workspace-scoped strings (`omega-forge-2026-05-25`)
- Session state stored in `/root/.openclaw/agents/main/sessions/`
- Constitutional session binding: `arif_session_init` anchors F1–F13 at session start
- No automatic session reset policy — managed by Arif or cron

**Hermes:**
- Session key format: `agent:main:{platform}:{chat_type}:{chat_id}`
- SQLite-backed SessionStore — survives restarts
- Configurable reset policies per platform:

| Policy | Default | Behavior |
|--------|---------|----------|
| `daily` | 4:00 AM | Reset at specific hour |
| `idle` | 1440 min | Reset after N minutes inactivity |
| `both` | combined | Whichever triggers first |

- Per-platform overrides supported (e.g., Telegram idle=240min, Discord idle=60min)
- `/new` or `/reset` for manual session reset
- Background sessions via `/background <prompt>` — isolated AIAgent, own session, result delivered to same chat

### 5. Authorization Model

**OpenClaw Gateway:**
- arifOS F11 sovereign signer — Ed25519 signature verification
- Telegram: authorized senders allowlist (267378578 = Arif)
- A2A mesh: Bearer token authentication (`aaa-a2a-token-dev`)
- No DM pairing — all authorization is explicit config

**Hermes Gateway:**
- Layered authorization evaluated in order:
  1. Per-platform allow-all (`TELEGRAM_ALLOW_ALL_USERS=true`)
  2. Platform allowlist (`TELEGRAM_ALLOWED_USERS=123456789,987654321`)
  3. DM pairing — unknown users get a pairing code, admin approves via CLI
  4. Global allow-all (`GATEWAY_ALLOW_ALL_USERS=true`)
  5. Default: **deny**
- DM pairing flow: `Admin: /pair` → Gateway gives code → User sends code → Admin approves with `hermes pairing approve telegram XKGH5N7P`
- Admin vs regular user split — admins get all slash commands; regular users get only explicitly whitelisted commands
- Pairing codes: 1-hour expiry, rate-limited, cryptographic randomness

### 6. Message Guard / Interrupt System

**OpenClaw:**
- No explicit "busy agent" guard described in its gateway design
- 888_HOLD system handles irreversible action gating — human confirmation required before execution
- Model swap / session interrupt via `/new` handled at the OpenClaw runtime level

**Hermes — Two-Level Guard:**

| Level | Location | What it does |
|-------|----------|--------------|
| Level 1 | `BaseAdapter` (`gateway/platforms/base.py`) | Checks `_active_sessions`. If session active → queue message + set interrupt event |
| Level 2 | `GatewayRunner` (`gateway/run.py`) | Checks `_running_agents`. Intercepts `/stop`, `/new`, `/queue`, `/status`, `/approve`, `/deny`. Everything else → `running_agent.interrupt()` |

**Busy input modes** (configurable):
- `interrupt` (default) — message interrupts, kills in-progress tool calls
- `queue` — follow-up waits, runs as next turn after current finishes
- `steer` — message injected via `/steer` arriving after next tool call (no new turn)

**Hermes Gateway** is a **messaging gateway** — its primary job is connecting an AIAgent to 20+ communication platforms with rich session management, authorization flows, and operational controls (circuit breakers, per-platform toolsets, busy-input modes). The intelligence lives in the AIAgent; the gateway is the connective tissue.

```
                              A2A mesh ← Hermes

                                       ↓

```

OpenClaw = **constitutional execution router**. Hermes = **messaging orchestration layer**. Different problems, different designs.
