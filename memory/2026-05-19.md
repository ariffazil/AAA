## MCP vs CLI: Flexible Tool Thesis (Tomaz Bratanic, Neo4j, May 18 2026)

**Core thesis:** MCP's "pile of narrow tools" is wrong shape for 2026 LLMs. A flexible CLI wins.

### 1. The Shape Shift
- Old world (2024): Models hallucinated flags, lost pipeline threads → narrow pre-baked tools were defensive necessity
- New world (2026): Models read --help on demand, know canonical CLIs from training, string bash together without supervision, retry on bad flags
- **"The hard part got easy, the easy part was always easy, all those neatly-wrapped tools mostly just bloat context for nothing."**

### 2. MCP Spec Allows Flexible Tools
- MCP spec only asks for typed tools, nothing more — says nothing about how narrow each tool must be
- Implementations gravitate toward many narrow tools for historic reasons, not because it's better for agents

### 3. Three Failure Modes of Narrow MCP Tools
- **Failure Mode A: Multi-environment context explosion** — 3 Neo4j instances = 12 schemas in context every turn. CLI: one binary, three profiles, zero per-turn context cost.
- **Failure Mode B: Chaining** — model becomes the pipe, intermediate results ride context verbatim. CLI: --params-from-stdin keeps intermediate data out of model context.
- **Failure Mode C: Cross-CLI pipe chains** — gh | jq | neo4j-cli in one pipeline, data flows through bash not agent, same token cost for 100 or 10,000 records.

### 4. Design Principle (Most Important)
- **"Read the terminal as MCP's limit case and design toward it: fewer tools, each one accepting expressive input, the agent doing the composing instead of you anticipating every combination in advance."**
- Servers that age well: smaller, more expressive surface on purpose

### 5. Caveat: Blast Radius
- Mitigations: sandbox, allowlist verbs, separate OS user, least-privilege credentials, read-only DB roles
- When you can't sandbox: MCP with small fixed surface still right answer

### 6. arifOS Alignment
- arifOS philosophy matches: expressive primitives over pre-baked narrow tools, agent composes
- **The gap the article doesn't close: governance under composition**
- Article: pre-flight safety via narrow surface
- arifOS: continuous safety via 888_JUDGE + VAULT999 + constitutional floors
- arifOS handles prompt injection + adversarial prompts better — model can't escape floor check even with flexible pipe access
